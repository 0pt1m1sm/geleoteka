## SaaS multi-tenancy

User intent: Geleoteka becomes a platform shipped to other specialist
auto-service shops (G-Class shop A, Porsche specialist B, ...) as a
SaaS solution where each tenant gets its own instance / domain. The
tenant chooses which modules they license (Service / Parts / Rentals /
CRM / Warehouse / ...).

What to design when this becomes a priority:

- `Tenant` model: id, name, domain(s), plan, `licensedModules: string[]`,
  brand assets, contact info. `licensedModules` enumerates only the
  **optional** verticals (`"service" | "parts" | "rentals"`);
  foundational modules (CRM, IAM, CMS, dashboard) are always on and
  not listed. Validation: `licensedModules.length ≥ 1` — an empty
  tenant has no business reason to exist. See
  `docs/research/2026-05-11-crm-vs-ops-overlap-policy.md` § Module
  taxonomy for the foundational/optional split.
- `tenantId` column on **every** major entity (User, Deal, RepairOrder,
  Part, ...). Add via a sweeping additive migration; backfill from the
  current "tenant of one"; enforce via DB-level row-level security or
  app-level filter at the Prisma extension layer.
- Custom domain → tenant resolution at the edge (Next.js middleware
  reads Host header → loads Tenant → injects into request context).
- Tenant-scoped admin UI: today's `/admin` is implicitly the one
  tenant; multi-tenancy switches this to `/admin` scoped to the
  resolved tenant. No URL change for the operator.
- Identity model: a User belongs to exactly one Tenant. Cross-tenant
  staff (e.g. an integrator login) is a separate `PlatformAdmin` model
  with its own routes (`/platform/*`).
- Module catalog: `lib/platform/modules.ts` declares all available
  modules with their nav metadata, route prefixes, server-action
  whitelists. Tenant's `licensedModules` filters this catalog at
  request time.

Migration risk is real (every query gains a tenant filter); should be
batched into a dedicated PRD with reversible steps.

## Role-aware sidebar + license-gated module groups

Sidebar today is one static list for every signed-in admin. Two
orthogonal axes need to filter it:

1. **Role** — MANAGER doesn't see Доступы; SALES (future) only sees
   CRM; ADMIN sees everything.
2. **License (optional modules only)** — if the tenant didn't license
   Rentals, the whole Rentals group hides in the sidebar and
   `/admin/rentals/*` 404s. Foundational groups (Дашборд, CRM,
   Доступы, Сайт) are always shown — they cannot be disabled.

Design sketch:

```ts
// lib/admin-nav.ts entry shape becomes:
type ModuleKey = "service" | "parts" | "rentals"; // optional only
interface AdminNavItem {
  href: string;
  label: string;
  requires?: { roles?: PermissionRole[]; module?: ModuleKey };
}
```

Server-side, the layout reads `session.permissionRole` and
`tenant.licensedModules`, filters `adminNav` before passing to
Sidebar / Header. Routes themselves use a `requireModule(moduleKey)`
guard (similar to current `requireRole`). Foundational groups have no
`module` requirement so they survive every filter.

Tradeoff: filter happens server-side per request — cheap. The added
complexity is the test matrix (role × license → expected nav).

Depends on SaaS multi-tenancy for the license axis. The role axis is
independent and can ship sooner.

## Other captured ideas

- **`Notification` → richer event stream.** Today `Notification` is a
  light fact-table of system-emitted messages (SMS sent, estimate
  ready). Could grow into an activity stream that subsumes
  `CommunicationLog` events too. Decide once CRM expansion Phase 2
  ships and we see real volume.
- **Auto-advance Deal.stage = DELIVERED** when all linked fulfillments
  are in terminal state (RO CLOSED, PartShipment DELIVERED,
  RentalBooking RETURNED). Phase 4 of Deal+Fulfillment PRD.
- **Per-tenant pricebook.** Today `Part.price` is global. SaaS pivot
  requires tenant-scoped prices. Out of scope for v1.
- **WhatsApp Business / Telegram bot inbound webhooks.** Schema is
  already messenger-agnostic via `CommunicationLog.channel`.
  Integration ships as a separate PRD when business need crystallizes.
- **Per-line approval inbox for clients.** Today clients accept whole
  estimates. Granular per-line accept/decline UX in the cabinet would
  unlock upsells (master proposes 3 jobs, customer picks 2). Phase 2+
  of CRM expansion.
