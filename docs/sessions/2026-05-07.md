### 4. createLocalStorageStore validator contract changed

Before: `validator?: (parsed: unknown) => T | null` — `null` ambiguously meant either "valid null value" or "self-heal".

After: `validator?: (parsed: unknown) => { ok: T } | null` — `{ ok: T }` is a valid value (T may include null without ambiguity); `null` is the unambiguous self-heal sentinel. The factory uses `if (result == null)` for defense against a buggy validator returning undefined.

Sole consumer (`lib/my-car-store.ts`) migrated: legitimate stored null wraps to `{ ok: null }`; invalid shapes return `null` (self-heal).

If you add a new `createLocalStorageStore` consumer:
- Without validator: parsed JSON is cast to T as-is (preserves the original BookingProvider/PartsCart behavior).
- With validator: must return `{ ok: T } | null`.

### 5. Type demotion convention

Lib types should NOT be `export`-ed unless one of these is true:
- Direct external import of the type.
- The type leaks via an exported function/value's signature (e.g., `setMyCar(car: MyCar | null)` makes `MyCar` reachable through inference).

`AdminNavGroup`, `AdminNavEntry`, `MyCar`, `YandexReviewsData`, `VehicleModel`, `BookingData` — kept exported (transitive leaks). `JWTPayload`, `SessionUser`, `YANDEX_ORG_ID`, `AdminNavLink` — demoted (zero external use, no signature leak).

When adding a new type to `lib/`: default to internal `interface Foo` / `type Foo`. Promote to `export` only when a consumer outside the file actually needs it.
