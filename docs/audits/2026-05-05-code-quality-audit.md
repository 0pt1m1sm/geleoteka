## Reuse over duplication

### Mobile/desktop nav primitive ‚Äî High

**Evidence:**
- `components/admin/AdminMobileNav.tsx:123-161` ‚Üî `components/shared/PanelMobileNav.tsx:88-121` (39 lines)
- `components/admin/AdminMobileNav.tsx:253-284` ‚Üî `components/admin/AdminSidebar.tsx:163-194` (32 lines)
- `components/shared/MobileMenu.tsx:52-75` ‚Üî `components/shared/PanelMobileNav.tsx:49-74` (24 lines)
- `components/admin/AdminMobileNav.tsx:129-152` ‚Üî `components/shared/PanelMobileNav.tsx:95-117` (24 lines)
- `components/admin/AdminMobileNav.tsx:46-68` ‚Üî `components/shared/PanelMobileNav.tsx:30-49` (23 lines)
- `components/admin/AdminMobileNav.tsx:41-63` ‚Üî `components/shared/PanelMobileNav.tsx:25-46` (23 lines)

**Rationale:** Six jscpd hits cluster around a single architectural duplication: the admin sidebar/mobile-nav and the shared mobile-menu/panel-nav each implement open-state tracking, focus management, escape handling, and link-list rendering separately. ~150 lines duplicated across 4 components ‚Äî every nav-related fix has to land in 4 places. An extractable `<NavPanel />` primitive in `components/shared/` taking a list shape + open-state owner as props would collapse this. Highest-leverage extraction in the codebase: visible from a maintenance angle (4 places to edit a nav rule today) and the reuse cost is structural (drift between admin and public nav UX is exactly what this kind of duplication produces).

### `defaultContact` server-side prefill pattern ‚Äî High

**Evidence:**
- `app/(public)/parts/cart/page.tsx` ‚Äî server component fetches `getSession()` ‚Üí builds `defaultContact = { name, phone, email }` ‚Üí passes to client form
- `app/(public)/booking/step-3/page.tsx` ‚Äî same shape, same getSession() ‚Üí defaultContact wiring ‚Üí client form prop

**Rationale:** Not flagged by jscpd (variable names + form fields differ enough), but PRD explicitly identified this as a duplication signal ‚Äî confirmed by manual file inspection. The pattern is "render a server-side form-prefill object from session and pass it to a client form via prop." With two sites today and any third checkout/booking-style flow tomorrow being a copy-paste, this earns a `lib/session-defaults.ts` helper with a `getDefaultContact()` server function. The session shape is also subject to drift; centralising the read makes future schema changes (add company name? loyalty tier?) one-edit, not N-edits.

### `useSyncExternalStore` cached-snapshot pattern ‚Äî High

**Evidence:**
- `lib/my-car-store.ts` ‚Äî `cachedRaw` + `cachedValue` + `getSnapshot()` returning the cached object when localStorage hasn't changed
- `components/booking/BookingProvider.tsx` ‚Äî same pattern for the booking wizard state
- `components/parts/PartsCart.tsx` ‚Äî same pattern for the cart cart-items array

**Rationale:** PRD called this out and confirmed via manual inspection. The pattern is mandated by the project's `geleoteka-conventions.md` (React 19 strict mode requires cached snapshots to avoid infinite re-render loops). Three implementations of the same boilerplate is the smell ‚Äî every new piece of localStorage state will want this primitive. A `lib/local-storage-store.ts` exposing `createLocalStorageStore<T>(key, initial, validator?)` returning `{ useStore, setStore }` would make the pattern one line of consumer code and prevent the next implementer from re-deriving it (and getting the cache key right). High leverage because it's both repeated and easy to get subtly wrong.

Medium because the gain is localised to the admin panel and the existing code isn't actively painful.

## Code quality

### Three dead exports in `lib/utils.ts` ‚Äî High

**Evidence:**
- `lib/utils.ts:5` ‚Äî `cn` (likely tailwind-merge classNames helper)
- `lib/utils.ts:40` ‚Äî `generateReferralCode`
- `lib/utils.ts:98` ‚Äî `getTierFromPoints`

**Rationale:** ts-prune flags all three as exported but with no consumer anywhere in the project (and no `(used in module)` annotation, so they're not used internally either). `cn` is the standard Tailwind classnames helper ‚Äî was either preempted during a previous refactor or never adopted. `generateReferralCode` and `getTierFromPoints` smell like loyalty-system scaffolding that didn't ship. High leverage because the fix is a one-PR delete (or, if you want to keep the helpers, find one consumer to wire them into); leaving dead helpers in `lib/utils.ts` makes that file harder to scan and gives the next implementer a misleading menu.

### `app/(cabinet)` ghost route group ‚Äî Medium

**Evidence:** 7 empty subdirectories under `app/(cabinet)/` (`cars`, `cars/add`, `history`, `loyalty`, `notifications`, `tracking`) with zero `.ts/.tsx` files. The actual portal lives entirely under `app/(portal)/cabinet/`.

**Rationale:** `(cabinet)` is the *route-group syntax* of an alternative portal layout that was started and never populated, then superseded by `(portal)/cabinet/`. The empty dir tree confuses readers (which is the real portal?), shows up in `find` output as if it had content, and clutters the route-group story. Fix is `rm -rf app/(cabinet)/`. Medium leverage because it's not actively breaking anything; just clutter.

### Three dead server actions ‚Äî Medium

**Evidence:**
- `app/actions/admin.ts:53` ‚Äî `assignMaster` (assigns a master user to a repair order ‚Äî no UI consumer)
- `app/actions/parts.ts:126` ‚Äî `deletePart` (soft-delete a part ‚Äî no UI button wired)
- `app/actions/supplier-orders.ts:106` ‚Äî `deleteSupplierOrder` (no UI button wired)

**Rationale:** Server actions exported without a UI consumer are a typical "feature half-shipped" smell ‚Äî either the admin UI was supposed to expose these and the buttons got dropped, or the actions were over-defined relative to actual product needs. For each: either wire it up (one DeleteButton in the relevant admin page) or delete the action. Medium leverage ‚Äî three small fixes, each independently bounded.

### `useMyCar` hook unused ‚Äî Medium

**Evidence:** `lib/my-car-store.ts:78` exports `useMyCar()` (returns the saved car as state). Only `setMyCar` (the setter) and `MY_CAR_KEY` are consumed elsewhere.

**Rationale:** The hook was provided to let any component subscribe to the saved-car selection, but the actual portal/parts UI uses `MyCarStrip` and `MyCarPicker` which read URL params and set localStorage via `setMyCar` ‚Äî the read path is via URL, not the hook. Either make the hook the canonical read API and migrate consumers, or delete it. Medium leverage ‚Äî an unused public API is an invitation for the next implementer to use it the wrong way.

### `lib/models-data.ts` re-export shim duplicates types ‚Äî Medium

**Evidence:** `lib/models-data.ts` re-exports `Generation`, `VehicleModel`, `Manufacturer`, `generationLabel`, `generationShort` from `./vehicle-catalog-types`. ts-prune flags 4 entries (`Generation`, `Manufacturer`, `generationShort` √ó 2 because both files export them).

**Rationale:** During the recent vehicle-catalog migration this file was reduced to a thin compatibility shim. Now grep confirms zero consumers still import from `@/lib/models-data` for type/helper purposes. Either delete `lib/models-data.ts` entirely OR keep it explicitly as a deprecation pointer. Medium leverage ‚Äî small structural cleanup that also reduces the surface area for "which file is the source of truth?" confusion.

### Internal-only types exported as public ‚Äî Low

**Evidence (9 entries flagged with `(used in module)`):**

- `lib/admin-nav.ts:15` ‚Äî `AdminNavLink`
- `lib/admin-nav.ts:28` ‚Äî `AdminNavEntry`
- `lib/auth.ts:9` ‚Äî `JWTPayload`
- `lib/auth.ts:14` ‚Äî `SessionUser` (this one IS used externally in many places ‚Äî likely a ts-prune false positive worth verifying)
- `lib/my-car-store.ts:5` ‚Äî `MyCar`
- `lib/yandex-reviews.ts:10` ‚Äî `YandexReviewsData`
- `lib/yandex.ts:2` ‚Äî `YANDEX_ORG_ID`
- `lib/vehicle-catalog.ts:6` ‚Äî `VehicleModel`
- `components/booking/BookingProvider.tsx:11` ‚Äî `BookingData`

**Rationale:** ts-prune marks these with `(used in module)` ‚Äî the symbol is used inside its own module but never imported by another file. Either the type/value is genuinely public-by-intention but unconsumed (a small surface-area expansion smell), or it should be `type Foo = ...` instead of `export type Foo = ...`. **Caveat:** ts-prune occasionally flags exports as "(used in module)" when they're actually used externally via path-alias imports it didn't follow ‚Äî `SessionUser` looks like a clear false positive on inspection. Hand-verify before deleting any of these. Low leverage ‚Äî pure hygiene.

### File length ‚Äî clean

**Evidence:** Largest file in scope is `prisma/seed-vehicles.ts` at 404 lines. All other files are under 400. Project rules flag >800 as a split signal; nothing approaches that.

**Rationale:** Recording this as a positive baseline. The codebase is disciplined on file length ‚Äî the largest files are seed data (linear, OK to be long) and a few rendered-page files that have multiple sections but no logic complexity. Nothing to action here.
