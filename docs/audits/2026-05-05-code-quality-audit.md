## Reuse over duplication

### Mobile/desktop nav primitive — High

**Evidence:**
- `components/admin/AdminMobileNav.tsx:123-161` ↔ `components/shared/PanelMobileNav.tsx:88-121` (39 lines)
- `components/admin/AdminMobileNav.tsx:253-284` ↔ `components/admin/AdminSidebar.tsx:163-194` (32 lines)
- `components/shared/MobileMenu.tsx:52-75` ↔ `components/shared/PanelMobileNav.tsx:49-74` (24 lines)
- `components/admin/AdminMobileNav.tsx:129-152` ↔ `components/shared/PanelMobileNav.tsx:95-117` (24 lines)
- `components/admin/AdminMobileNav.tsx:46-68` ↔ `components/shared/PanelMobileNav.tsx:30-49` (23 lines)
- `components/admin/AdminMobileNav.tsx:41-63` ↔ `components/shared/PanelMobileNav.tsx:25-46` (23 lines)

**Rationale:** Nine jscpd hits cluster around a single architectural duplication: the admin sidebar/mobile-nav and the shared mobile-menu/panel-nav each implement open-state tracking, focus management, escape handling, and link-list rendering separately. ~210 jscpd-reported lines duplicated across the 4 components (some clone ranges overlap, so the unique extractable block is somewhat smaller — but every nav-related fix has to land in 4 places regardless). An extractable `<NavPanel />` primitive in `components/shared/` taking a list shape + open-state owner as props would collapse this. Highest-leverage extraction in the codebase: visible from a maintenance angle (4 places to edit a nav rule today) and the reuse cost is structural (drift between admin and public nav UX is exactly what this kind of duplication produces).

### `defaultContact` server-side prefill pattern — High

**Evidence:**
- `app/(public)/parts/cart/page.tsx` — server component fetches `getSession()` → builds `defaultContact = { name, phone, email }` → passes to client form
- `app/(public)/booking/step-3/page.tsx` — same shape, same getSession() → defaultContact wiring → client form prop

**Rationale:** Not flagged by jscpd (variable names + form fields differ enough), but PRD explicitly identified this as a duplication signal — confirmed by manual file inspection. The pattern is "render a server-side form-prefill object from session and pass it to a client form via prop." With two sites today and any third checkout/booking-style flow tomorrow being a copy-paste, this earns a `lib/session-defaults.ts` helper with a `getDefaultContact()` server function. The session shape is also subject to drift; centralising the read makes future schema changes (add company name? loyalty tier?) one-edit, not N-edits.

### `useSyncExternalStore` cached-snapshot pattern — High

**Evidence:**
- `lib/my-car-store.ts` — `cachedRaw` + `cachedValue` + `getSnapshot()` returning the cached object when localStorage hasn't changed
- `components/booking/BookingProvider.tsx` — same pattern for the booking wizard state
- `components/parts/PartsCart.tsx` — same pattern for the cart cart-items array

**Rationale:** PRD called this out and confirmed via manual inspection. The pattern is mandated by the project's `geleoteka-conventions.md` (React 19 strict mode requires cached snapshots to avoid infinite re-render loops). Three implementations of the same boilerplate is the smell — every new piece of localStorage state will want this primitive. A `lib/local-storage-store.ts` exposing `createLocalStorageStore<T>(key, initial, validator?)` returning `{ useStore, setStore }` would make the pattern one line of consumer code and prevent the next implementer from re-deriving it (and getting the cache key right). High leverage because it's both repeated and easy to get subtly wrong.

### Admin form shells (PartForm/PartEditForm, RentalEditForm/SupplierEditForm) — Medium

**Evidence:**
- `components/admin/PartEditForm.tsx:29-41` ↔ `components/admin/PartForm.tsx:13-25` (13 lines — `useActionState` + state plumbing)
- `components/admin/PartEditForm.tsx:46-57` ↔ `components/admin/PartForm.tsx:30-41` (12 lines — error banner + section header)
- `components/admin/RentalEditForm.tsx:33-45` ↔ `components/admin/SupplierEditForm.tsx:26-38` (13 lines — top-of-form chrome)

**Rationale:** Admin CRUD pairs (create + edit per entity) share the form scaffold but differ in field set. The `useActionState` plumbing, error-banner positioning, and "title + description" header pair are repeated structurally. Extracting an `<AdminFormShell title={...} action={...} error={state.error}>{children}</AdminFormShell>` would let each form focus on its fields. Medium because the gain is localised to the admin panel and the existing code isn't actively painful.

## Code quality

### Three dead exports in `lib/utils.ts` — High

**Evidence:**
- `lib/utils.ts:5` — `cn` (likely tailwind-merge classNames helper)
- `lib/utils.ts:40` — `generateReferralCode`
- `lib/utils.ts:98` — `getTierFromPoints`

**Rationale:** ts-prune flags all three as exported but with no consumer anywhere in the project (and no `(used in module)` annotation, so they're not used internally either). `cn` is the standard Tailwind classnames helper — was either preempted during a previous refactor or never adopted. `generateReferralCode` and `getTierFromPoints` smell like loyalty-system scaffolding that didn't ship. High leverage because the fix is a one-PR delete (or, if you want to keep the helpers, find one consumer to wire them into); leaving dead helpers in `lib/utils.ts` makes that file harder to scan and gives the next implementer a misleading menu.

### `app/(cabinet)` ghost route group — Medium

**Evidence:** 7 empty subdirectories under `app/(cabinet)/` (`cars`, `cars/add`, `history`, `loyalty`, `notifications`, `tracking`) with zero `.ts/.tsx` files. The actual portal lives entirely under `app/(portal)/cabinet/`.

**Rationale:** `(cabinet)` is the *route-group syntax* of an alternative portal layout that was started and never populated, then superseded by `(portal)/cabinet/`. The empty dir tree confuses readers (which is the real portal?), shows up in `find` output as if it had content, and clutters the route-group story. Fix is `rm -rf app/(cabinet)/`. Medium leverage because it's not actively breaking anything; just clutter.

### Three dead server actions — Medium

**Evidence:**
- `app/actions/admin.ts:53` — `assignMaster` (assigns a master user to a repair order — no UI consumer)
- `app/actions/parts.ts:126` — `deletePart` (soft-delete a part — no UI button wired)
- `app/actions/supplier-orders.ts:106` — `deleteSupplierOrder` (no UI button wired)

**Rationale:** Server actions exported without a UI consumer are a typical "feature half-shipped" smell — either the admin UI was supposed to expose these and the buttons got dropped, or the actions were over-defined relative to actual product needs. For each: either wire it up (one DeleteButton in the relevant admin page) or delete the action. Medium leverage — three small fixes, each independently bounded.

### `useMyCar` hook unused — Medium

**Evidence:** `lib/my-car-store.ts:78` exports `useMyCar()` (returns the saved car as state). Only `setMyCar` (the setter) and `MY_CAR_KEY` are consumed elsewhere.

**Rationale:** The hook was provided to let any component subscribe to the saved-car selection, but the actual portal/parts UI uses `MyCarStrip` and `MyCarPicker` which read URL params and set localStorage via `setMyCar` — the read path is via URL, not the hook. Either make the hook the canonical read API and migrate consumers, or delete it. Medium leverage — an unused public API is an invitation for the next implementer to use it the wrong way.

### `lib/models-data.ts` re-export shim has partially-dead exports — Medium

**Evidence:** `lib/models-data.ts` re-exports `Generation`, `VehicleModel`, `Manufacturer`, `generationLabel`, `generationShort` from `./vehicle-catalog-types`. ts-prune flags 3 of those re-exports as unused: `Generation` (line 13), `Manufacturer` (line 13), `generationShort` (line 14). The corresponding 3 source exports in `lib/vehicle-catalog-types.ts` (re-exported through `lib/vehicle-catalog.ts`) are also flagged. `VehicleModel` and `generationLabel` are NOT flagged — they are actively imported by `components/booking/Step1ServiceVehicle.tsx:6` and `components/parts/MyCarPicker.tsx:5`.

**Rationale:** During the recent vehicle-catalog migration this file was reduced to a thin compatibility shim. Two of the five re-exports (`VehicleModel`, `generationLabel`) are still consumed by client components — those imports are the reason `lib/models-data.ts` cannot simply be deleted. The other three (`Generation`, `Manufacturer`, `generationShort`) re-export through both `models-data.ts` and `vehicle-catalog.ts` but are never imported anywhere. Recommend partial cleanup: drop the unused re-exports from both files, keep the ones with active consumers (or migrate the two consumers to import directly from `@/lib/vehicle-catalog-types` and then delete `lib/models-data.ts` entirely). Medium leverage — small structural cleanup that also reduces the surface area for "which file is the source of truth?" confusion.

### Internal-only types exported as public — Low

**Evidence (9 entries flagged with `(used in module)`):**

- `lib/admin-nav.ts:15` — `AdminNavLink`
- `lib/admin-nav.ts:28` — `AdminNavEntry`
- `lib/auth.ts:9` — `JWTPayload`
- `lib/auth.ts:14` — `SessionUser` (verified: only used internally as the return type of `getSession()`/`requireAuth()`/`requireRole()`; no external file imports the type itself, only calls the functions and uses inferred types)
- `lib/my-car-store.ts:5` — `MyCar`
- `lib/yandex-reviews.ts:10` — `YandexReviewsData`
- `lib/yandex.ts:2` — `YANDEX_ORG_ID`
- `lib/vehicle-catalog.ts:6` — `VehicleModel` (re-exported through `lib/models-data.ts` and consumed via that re-export — see "models-data shim" finding above for context)
- `components/booking/BookingProvider.tsx:11` — `BookingData`

**Rationale:** ts-prune marks these with `(used in module)` — the symbol is used inside its own module but never imported by another file directly. Each is a candidate to demote from `export type Foo = ...` to `type Foo = ...` (or `export function` to `function` for `YANDEX_ORG_ID`-style values). Hand-verify each before changing — for some (`VehicleModel`) the consumer reaches the type indirectly via a re-export shim. Low leverage — pure hygiene.

### File length — clean

**Evidence:** Largest file in scope is `prisma/seed-vehicles.ts` at 404 lines. All other files are under 400. Project rules flag >800 as a split signal; nothing approaches that.

**Rationale:** Recording this as a positive baseline. The codebase is disciplined on file length — the largest files are seed data (linear, OK to be long) and a few rendered-page files that have multiple sections but no logic complexity. Nothing to action here.
